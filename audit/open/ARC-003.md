# Architecture Finding: ARC-003

## Singleton Overuse in DI

**Severity:** MEDIUM  
**Impact:** State leakage, testing difficulties, memory growth  
**Status:** 🔴 **OPEN**

---

## Description

All services registered as singleton, even stateful ones.

### Current Pattern

```v
sr.register_logger_service('info')      // Singleton - OK (stateless)
sr.register_cache_service(1000)         // Singleton - OK (shared cache)
sr.register_validation_service()        // Singleton - Should be transient
sr.register_metrics_service()           // Singleton - Should be scoped
sr.register_auth_service(3600)          // Singleton - Should be scoped
```

---

## Impact

- Stateful services live forever
- Test isolation requires manual cleanup
- Memory grows unbounded
- No request-scoped state management

---

## Remediation

Use appropriate lifecycles:

```v
// Stateless - Singleton
sr.register_logger_service('info')

// Shared cache - Singleton  
sr.register_cache_service(1000)

// Stateless, no shared state - Transient
container.register_transient('validation', fn () voidptr {
    return new_validation_service()
})

// Per-request state - Scoped
container.register_scoped('metrics', fn () voidptr {
    return new_metrics_service()
})

// Per-session state - Scoped
container.register_scoped('auth', fn () voidptr {
    return new_auth_service(3600)
})
```
